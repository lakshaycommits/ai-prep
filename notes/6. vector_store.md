# RAG Vector Stores — Interview Notes

## What a vector store actually is

A vector store is a database optimized for one specific operation: given a query vector, find the K most similar vectors from a stored collection of millions or billions of vectors — fast.

A regular database (Postgres, MySQL) can store vectors as arrays and compute cosine similarity, but it does this by scanning every row — O(n) per query. At 10M vectors this is unusably slow. Vector stores solve this with specialized indexing algorithms that make similarity search sub-linear.

The core problem: **exact nearest neighbor search is O(n×d)** where n = corpus size and d = dimensions. At n=10M and d=1536, that's 15.36 billion multiplications per query. Vector stores use **Approximate Nearest Neighbor (ANN)** algorithms to trade a small amount of accuracy for orders-of-magnitude speed improvement.

---

## Exact vs Approximate Nearest Neighbor

### Exact nearest neighbor (brute force)

Computes similarity between the query and every stored vector. Returns the true top-K.

```python
import numpy as np

def exact_search(query_vec, corpus_vecs, k=5):
    scores = np.dot(corpus_vecs, query_vec)  # O(n×d)
    top_k = np.argsort(scores)[-k:][::-1]
    return top_k
```

- **Accuracy:** 100% — always finds the true nearest neighbors
- **Speed:** O(n×d) — linear in corpus size, unusable at scale
- **Use case:** Small corpora (<100K vectors), offline batch jobs, ground truth for eval

### Approximate nearest neighbor (ANN)

Builds an index structure at index time that allows skipping large portions of the corpus at query time. Returns the approximate top-K — usually 95–99% accurate, 100–1000x faster.

- **Accuracy:** 95–99% (tunable via recall parameter)
- **Speed:** O(log n) to O(√n) depending on algorithm
- **Use case:** Any production system with >100K vectors

The accuracy loss is acceptable because: the 1–5% of "missed" nearest neighbors are usually semantically very close to the ones that were found. Missing the 4th closest vector when you retrieve top-5 is rarely a problem in practice.

## HNSW — the algorithm powering modern vector stores

HNSW (Hierarchical Navigable Small World) is the dominant ANN algorithm used by Qdrant, Weaviate, Pinecone, and most production vector stores. Understanding it is the key technical depth interviewers probe.

### The intuition

HNSW builds a multi-layer graph where:
- **Bottom layer:** contains all vectors, densely connected to neighbors
- **Upper layers:** contain progressively fewer vectors, longer-range connections
- **Search:** starts at the top (sparse, fast navigation), descends to the bottom (dense, precise)

Think of it like navigating a city: highways (top layers) get you close quickly, local streets (bottom layer) get you to the exact destination.

### The graph structure

```
Layer 2 (sparse):    A ──────────────────── F
                      \                    /
Layer 1 (medium):     A ──── C ──── E ── F
                       \   /   \   /
Layer 0 (dense):       A─B─C─D─E─F─G─H─I
                       all vectors, short-range connections
```

Each vector exists in the bottom layer. It probabilistically appears in higher layers — roughly 1/e ≈ 37% chance per level up. So most vectors only exist at layer 0, fewer at layer 1, very few at layer 2+.

### Search algorithm (greedy best-first)

```
1. Enter at top layer at the single entry point
2. Greedily move to the neighbor closest to query vector
3. When no neighbor is closer than current node → drop to next layer
4. Repeat until bottom layer
5. At bottom layer: explore neighborhood thoroughly, collect top-K candidates
```

```python
# Conceptual HNSW search (not actual implementation)
def hnsw_search(query_vec, entry_point, layers, k=5):
    current = entry_point
    
    # traverse upper layers greedily
    for layer in reversed(layers[1:]):
        current = greedy_search(query_vec, current, layer, ef=1)
    
    # thorough search at bottom layer
    candidates = greedy_search(query_vec, current, layers[0], ef=ef_search)
    
    return top_k(candidates, k)
```

### The two critical HNSW parameters

**`m` (number of connections per node):**
- How many bidirectional edges each vector has in the graph
- Higher m → better recall, more memory, slower index build
- Typical values: 8–64
- Qdrant default: 16

```python
# In Qdrant
from qdrant_client.models import HnswConfigDiff

client.create_collection(
    collection_name="docs",
    vectors_config=VectorParams(size=1536, distance=Distance.COSINE),
    hnsw_config=HnswConfigDiff(
        m=16,              # connections per node
        ef_construct=100,  # candidates considered during index build
    )
)
```

**`ef` (exploration factor at query time):**
- How many candidates are explored at the bottom layer during search
- Higher ef → better recall, higher latency
- Must be ≥ k (top-K you want to retrieve)
- Typical values: 50–200

```python
# At query time
results = qdrant.search(
    collection_name="docs",
    query_vector=query_vector,
    limit=5,
    search_params=SearchParams(hnsw_ef=128)  # higher = better recall, slower
)
```

### HNSW trade-off table

| Parameter | Increase effect | Decrease effect |
|---|---|---|
| `m` | Better recall, more memory | Worse recall, less memory |
| `ef_construct` | Better index quality, slower build | Faster build, worse index |
| `ef` (search) | Better recall, higher latency | Lower latency, worse recall |

### Memory footprint

```python
# HNSW memory estimate for Qdrant
n_vectors = 1_000_000
dim = 1536
m = 16
bytes_per_float = 4

vector_memory = n_vectors * dim * bytes_per_float          # 6.14 GB
graph_memory  = n_vectors * m * 2 * 8                      # ~0.26 GB (links)
total_memory  = vector_memory + graph_memory               # ~6.4 GB
```

## Qdrant architecture deep dive

Since you use Qdrant in production (Synapse), this section connects internals to what you've already built.

### Core concepts

**Collection:** The top-level namespace. Equivalent to a table in SQL. One collection = one vector space with consistent dimensions and distance metric.

**Point:** A single unit — one vector + one payload (metadata dict) + one ID.

**Segment:** Qdrant's internal storage unit. A collection is split into multiple segments. Each segment has its own HNSW index. Segments are merged/optimized in the background.

**Shard:** For distributed deployments, a collection is split into shards across nodes. Each shard contains a subset of points.

```
Collection "docs"
├── Shard 0 (node 1)
│   ├── Segment A (indexed, immutable)
│   ├── Segment B (indexed, immutable)
│   └── Segment C (fresh, being indexed)
└── Shard 1 (node 2)
    ├── Segment D
    └── Segment E
```

### Write path

```
upsert(points)
    ↓
WAL (Write-Ahead Log) — durability guarantee
    ↓
In-memory buffer (fresh segment, no HNSW yet)
    ↓
Background optimizer triggers when segment is large enough
    ↓
HNSW index built on segment
    ↓
Segment becomes immutable, searchable via HNSW
```

Fresh points (not yet indexed) are searched via brute force in the buffer. This is why very recent upserts might have slightly lower recall — they're not in HNSW yet. Qdrant handles this transparently.

### Named vectors (multiple vector spaces per collection)

```python
# For hybrid search: store dense + sparse vectors on same point
client.create_collection(
    collection_name="docs",
    vectors_config={
        "dense": VectorParams(size=1536, distance=Distance.COSINE),
        "sparse": SparseVectorParams(index=SparseIndexParams())
    }
)

# Upsert with both
client.upsert(
    collection_name="docs",
    points=[PointStruct(
        id=1,
        vector={
            "dense": [0.1, 0.2, ...],
            "sparse": SparseVector(indices=[42, 891], values=[0.6, 0.3])
        },
        payload={"text": "...", "source": "policy_doc"}
    )]
)
```

### Payload indexing

Without a payload index, metadata filters scan all payloads linearly — O(n). With an index, filter lookups are O(log n).

```python
# Always create payload indexes on fields you filter by
client.create_payload_index(
    collection_name="docs",
    field_name="tenant_id",
    field_schema=PayloadSchemaType.KEYWORD   # exact match
)

client.create_payload_index(
    collection_name="docs",
    field_name="created_at",
    field_schema=PayloadSchemaType.DATETIME  # range queries
)

client.create_payload_index(
    collection_name="docs",
    field_name="chunk_type",
    field_schema=PayloadSchemaType.KEYWORD
)
```

### Quantization in Qdrant

Qdrant supports three quantization modes to reduce memory:

```python
from qdrant_client.models import (
    ScalarQuantizationConfig, ProductQuantizationConfig,
    BinaryQuantizationConfig, QuantizationType
)

# Scalar quantization: float32 → int8 (4x memory reduction, ~1% quality loss)
ScalarQuantizationConfig(type=QuantizationType.INT8, quantile=0.99, always_ram=True)

# Binary quantization: float32 → 1 bit (32x memory reduction, ~5-10% quality loss)
# Best combined with re-scoring
BinaryQuantizationConfig(always_ram=True)

# Product quantization: highest compression, more quality loss
ProductQuantizationConfig(compression=Compression.X16, always_ram=True)
```

**`always_ram=True`:** Keep quantized vectors in RAM even if full vectors are on disk. Fast approximate search in RAM, re-score with full vectors on disk. Best latency/memory balance.

## Vector store comparison

| Store | Type | Best for | ANN Algorithm | Hybrid search | Notes |
|---|---|---|---|---|---|
| **Qdrant** | Self-hosted / Cloud | Performance, filtering, Rust-based | HNSW | Yes (native) | Best filtering capabilities, payload indexes |
| **Pinecone** | Fully managed | Simplest production setup | Proprietary | Yes | No self-hosting, vendor lock-in |
| **Weaviate** | Self-hosted / Cloud | Hybrid search, GraphQL API | HNSW | Yes (BM25 built-in) | Schema-based, heavier setup |
| **ChromaDB** | Self-hosted | Local dev, prototyping | HNSW | No | Not for production at scale |
| **pgvector** | Postgres extension | Existing Postgres infra | HNSW / IVFFlat | No (manual) | Convenient but slower at scale |
| **FAISS** | Library (not a DB) | Research, offline batch | HNSW, IVFFlat, PQ | No | No persistence, no server, Meta's library |
| **Azure AI Search** | Managed (Azure) | Azure-native deployments | HNSW | Yes (built-in) | Integrates with Azure OpenAI |
| **Milvus** | Self-hosted / Cloud | Large-scale (billions) | HNSW, IVF, DiskANN | Yes | Most feature-rich, complex ops |

### When to pick each

**Qdrant:** You need strong metadata filtering, hybrid search, and want self-hosting. Best performance/feature ratio. Your current choice for Synapse — correct call.

**Pinecone:** Team has no infra capacity and wants fully managed. Simplest API, but expensive at scale and no self-hosting.

**Weaviate:** Need GraphQL API or tight schema enforcement on your documents. Good hybrid search out of the box.

**pgvector:** Already running Postgres and corpus is <1M vectors. Don't want to add another service. Acceptable performance at small scale.

**ChromaDB:** Local development and prototyping only. Never prod.

**FAISS:** Offline batch processing, research, or when you need maximum control over the ANN algorithm with no server overhead.

**Azure AI Search:** Azure stack and want managed service that integrates with Azure OpenAI, Cognitive Services.

**Milvus:** Billions of vectors, need distributed scale beyond what Qdrant handles comfortably.

## IVFFlat — the other common ANN algorithm

HNSW dominates, but IVFFlat (Inverted File Index with Flat re-scoring) is used in FAISS and older pgvector setups. Worth knowing for interviews.

### How it works

```
Index time:
1. Cluster all vectors into K centroids (K-means clustering)
2. Each vector assigned to its nearest centroid
3. Inverted index: centroid → list of vector IDs

Query time:
1. Find the nprobe nearest centroids to the query vector
2. Retrieve all vectors in those clusters
3. Brute-force search within those clusters
4. Return top-K
```

```python
# FAISS IVFFlat example
import faiss
import numpy as np

d = 1536          # dimensions
n_clusters = 1000 # number of Voronoi cells

quantizer = faiss.IndexFlatL2(d)
index = faiss.IndexIVFFlat(quantizer, d, n_clusters)

# train on representative sample (required for IVFFlat, not HNSW)
index.train(corpus_vectors)
index.add(corpus_vectors)

index.nprobe = 10  # search 10 nearest clusters at query time
distances, indices = index.search(query_vector, k=5)
```

### HNSW vs IVFFlat

| | HNSW | IVFFlat |
|---|---|---|
| Build time | Slower | Faster |
| Memory | Higher (graph links) | Lower |
| Query speed | Fast | Fast |
| Recall | Higher | Lower (depends on nprobe) |
| Requires training | No | Yes (K-means on corpus) |
| Dynamic inserts | Easy | Requires re-training at scale |
| Used in | Qdrant, Weaviate, Pinecone | FAISS, older pgvector |

HNSW is preferred for dynamic corpora (frequent inserts). IVFFlat is preferred when memory is extremely tight.

## DiskANN — for billion-scale corpora

When corpus exceeds GPU/CPU RAM (billions of vectors), DiskANN stores vectors on SSD and uses graph-based traversal that minimizes random disk reads.

- Used by Microsoft Azure AI Search at scale
- Milvus supports DiskANN
- Qdrant has experimental on-disk HNSW support

Not commonly asked in interviews unless targeting large-scale search roles.

## Practical Qdrant patterns for production

### Collection creation with all options

```python
from qdrant_client import QdrantClient
from qdrant_client.models import (
    VectorParams, Distance, HnswConfigDiff,
    ScalarQuantizationConfig, QuantizationType, OptimizersConfigDiff
)

client = QdrantClient(url="http://localhost:6333")

client.create_collection(
    collection_name="indigo_docs",
    vectors_config=VectorParams(
        size=1536,
        distance=Distance.COSINE,
        on_disk=False   # keep in RAM for speed
    ),
    hnsw_config=HnswConfigDiff(
        m=16,
        ef_construct=100,
        on_disk=False   # HNSW graph in RAM
    ),
    quantization_config=ScalarQuantizationConfig(
        type=QuantizationType.INT8,
        quantile=0.99,
        always_ram=True  # quantized vectors in RAM, full vectors optionally on disk
    ),
    optimizers_config=OptimizersConfigDiff(
        indexing_threshold=20000  # build HNSW after 20K vectors in segment
    )
)
```

### Batch upsert pattern

```python
from qdrant_client.models import PointStruct
import uuid

def batch_upsert(chunks: list[dict], vectors: list, batch_size: int = 100):
    for i in range(0, len(chunks), batch_size):
        batch_chunks = chunks[i:i+batch_size]
        batch_vecs = vectors[i:i+batch_size]
        
        points = [
            PointStruct(
                id=str(uuid.uuid4()),
                vector=vec.tolist(),
                payload={
                    "text": chunk["content"],
                    "source": chunk["source"],
                    "tenant_id": chunk["tenant_id"],
                    "chunk_index": chunk["index"],
                    "created_at": chunk["created_at"]
                }
            )
            for chunk, vec in zip(batch_chunks, batch_vecs)
        ]
        
        client.upsert(collection_name="indigo_docs", points=points)
```

### Scroll for full corpus operations

```python
# Never use search with high limit for full corpus scan
# Use scroll instead — pagination over all points

offset = None
all_points = []

while True:
    result, next_offset = client.scroll(
        collection_name="indigo_docs",
        limit=1000,
        offset=offset,
        with_payload=True,
        with_vectors=False   # don't fetch vectors unless needed
    )
    all_points.extend(result)
    
    if next_offset is None:
        break
    offset = next_offset
```

## Failure modes

| Symptom | Likely cause | Fix |
|---|---|---|
| Recall degrades as corpus grows | ef too low for corpus size | Increase ef at query time |
| Filtering is slow | No payload index on filter field | Create payload index |
| Memory OOM at scale | Full float32 vectors in RAM | Enable quantization + on_disk |
| Recent upserts not searchable | Still in write buffer, not indexed | Wait for segment optimization or force optimize |
| Query latency high under load | HNSW graph not in RAM | Set on_disk=False for hnsw_config |
| Recall much lower than expected | ef_construct too low at build time | Rebuild collection with higher ef_construct |
| Cross-tenant data leakage | No tenant_id filter on query | Enforce filter in retrieval layer, never optional |

## Key interview questions + strong answers

**Q: "What is ANN and why do we need it?"**
> Exact nearest neighbor search is O(n×d) — at 10M vectors and 1536 dimensions that's 15 billion multiplications per query, completely unscalable. ANN algorithms like HNSW build graph or cluster-based index structures at index time so search is O(log n) instead of O(n), at the cost of returning approximate rather than exact nearest neighbors — typically 95–99% accurate. In practice the accuracy loss is negligible because the missed neighbors are always semantically very close to the ones returned.

**Q: "Explain HNSW."**
> HNSW builds a multi-layer graph over the vector corpus. Upper layers have fewer vectors with long-range connections — like highways. Lower layers have all vectors with short-range connections — like local streets. Search starts at the top layer and greedily navigates toward the query, then descends to lower layers for refinement. The two key parameters are m (connections per node, controls recall vs memory) and ef (exploration factor at query time, controls recall vs latency). Higher m and ef give better recall but cost more memory and latency respectively.

**Q: "How would you scale a RAG system to 100M documents?"**
> First, reduce vector memory with quantization — int8 gives 4x reduction with ~1% quality loss, binary gives 32x reduction with re-scoring. Use MRL-truncated embeddings (256–512 dims) to reduce vector size further. Distribute across multiple Qdrant shards — Qdrant supports horizontal sharding natively. Put quantized vectors in RAM and full vectors on NVMe SSD for re-scoring. Payload index every filter field. Consider DiskANN if vectors don't fit in RAM even after quantization.

**Q: "When would you use pgvector instead of Qdrant?"**
> When the corpus is under ~1M vectors and I'm already running Postgres — the convenience of one less service outweighs the performance difference. pgvector's HNSW is slower than Qdrant's at scale, but at small corpora the difference is negligible. The moment I need strong metadata filtering, hybrid search, or plan to grow beyond 1M vectors, I'd switch to Qdrant. pgvector also lacks quantization support, so memory becomes a problem faster.

## Key terms for interviews

- **ANN (Approximate Nearest Neighbor)** — sub-linear similarity search; trades small accuracy loss for massive speed gain
- **HNSW** — dominant ANN algorithm; multi-layer navigable graph; used by Qdrant, Weaviate, Pinecone
- **m** — HNSW connections per node; controls recall vs memory
- **ef_construct** — HNSW build-time exploration factor; controls index quality vs build speed
- **ef** — HNSW query-time exploration factor; controls recall vs latency
- **IVFFlat** — cluster-based ANN; used in FAISS; requires training step; lower recall than HNSW
- **Segment** — Qdrant's internal storage unit; each has its own HNSW index
- **Shard** — distributes a collection across nodes for horizontal scaling
- **Payload index** — index on a metadata field; required for O(log n) filtering
- **Scalar quantization** — float32 → int8; 4x memory reduction, ~1% quality loss
- **Binary quantization** — float32 → 1 bit; 32x memory reduction; needs re-scoring
- **always_ram** — Qdrant option to keep quantized vectors in RAM for fast approximate search
- **Named vectors** — multiple vector spaces per Qdrant point; used for dense + sparse hybrid
- **WAL (Write-Ahead Log)** — Qdrant's durability mechanism; writes go to WAL before indexing
