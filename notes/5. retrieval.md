# RAG Retrieval Strategies — Interview Notes

## The retrieval problem in one line

Given a user query, find the K most relevant chunks from a corpus of millions. Every strategy below is a different answer to "what does relevant mean and how do we find it efficiently."

---

## Strategy 1: Basic semantic search (cosine similarity)

The baseline. Embed the query, compute cosine similarity against all stored vectors, return top-K.

```python
query_vector = embedding_model.encode("refund policy for cancelled flights")

results = qdrant.search(
    collection_name="docs",
    query_vector=query_vector,
    limit=5
)
```

**What it gets right:** Vocabulary-independent matching. "refund" matches "money back", "compensation" matches "reimbursement" — because semantic meaning is close in vector space.

**What it misses:** Exact-match queries. If someone searches "PNR 4X7K9", semantic search struggles — it has no concept that this is a specific identifier. The embedding of "PNR 4X7K9" is geometrically close to other PNR-like strings, but your corpus may have it as an exact string. Keyword search handles this trivially; semantic search does not.

This is why basic semantic search alone is not enough for production systems.

## Strategy 2: Hybrid search (BM25 + Vector + RRF)

### What BM25 is

BM25 (Best Match 25) is a keyword-based ranking algorithm. It scores documents based on term frequency and inverse document frequency — how often your query terms appear in the document, weighted by how rare those terms are across the whole corpus.

```
BM25(query, doc) = Σ IDF(term) × TF(term, doc) × (k1 + 1) / (TF + k1 × (1 - b + b × |doc|/avgdl))
```

You don't need to memorize the formula. What matters:
- High score if query terms appear frequently in the document
- High score if query terms are rare across the corpus (IDF)
- Penalizes very long documents (length normalization via `b`)
- Pure keyword matching — no semantic understanding

**BM25 is good at:** Exact matches, product codes, names, IDs, rare technical terms.
**BM25 is bad at:** Synonyms, paraphrasing, conceptual queries without exact word overlap.

**Vector search is good at:** Semantic similarity, paraphrasing, concept matching.
**Vector search is bad at:** Exact matches, rare tokens, out-of-vocabulary terms.

They fail in opposite scenarios. Combining them covers both.

### Reciprocal Rank Fusion (RRF)

The fusion algorithm that merges BM25 and vector search result lists.

```
RRF_score(doc) = Σ 1 / (k + rank_in_list)
```

Where `k` is a constant (typically 60) that dampens the impact of very high rankings.

**How it works step by step:**

```
Query: "PNR 4X7K9 refund status"

BM25 results:          Vector results:
1. Doc A (PNR match)   1. Doc C (refund policy)
2. Doc B               2. Doc A
3. Doc C               3. Doc D
4. Doc D               4. Doc B

RRF scores:
Doc A: 1/(60+1) + 1/(60+2) = 0.01639 + 0.01613 = 0.03252  ← appears in both, ranked high
Doc C: 1/(60+3) + 1/(60+1) = 0.01587 + 0.01639 = 0.03226
Doc B: 1/(60+2) + 1/(60+4) = 0.01613 + 0.01563 = 0.03175
Doc D: 1/(60+4) + 1/(60+3) = 0.01563 + 0.01587 = 0.03149

Final ranking: A, C, B, D
```

Documents appearing high in **both** lists get boosted. A document that ranks #1 in BM25 but doesn't appear in vector results still contributes its BM25 score. RRF is robust — it doesn't require score normalization across the two systems.

### Implementation

```python
from qdrant_client import QdrantClient
from qdrant_client.models import SparseVector

# Qdrant supports hybrid search natively with sparse + dense vectors
# Sparse vector = BM25-style representation
# Dense vector = embedding model output

# Index time: store both sparse and dense
client.upsert(
    collection_name="docs",
    points=[
        PointStruct(
            id=chunk_id,
            vector={
                "dense": dense_embedding,         # from BGE/OpenAI
                "sparse": SparseVector(           # from BM25/SPLADE
                    indices=[101, 4502, 891],
                    values=[0.42, 0.31, 0.18]
                )
            },
            payload={"text": chunk_text, "source": "policy_doc"}
        )
    ]
)

# Query time: search both, fuse with RRF
from qdrant_client.models import Prefetch, FusionQuery, Fusion

results = client.query_points(
    collection_name="docs",
    prefetch=[
        Prefetch(query=dense_vector, using="dense", limit=50),
        Prefetch(query=sparse_vector, using="sparse", limit=50),
    ],
    query=FusionQuery(fusion=Fusion.RRF),
    limit=5
)
```

### Why RRF over weighted average?

Weighted average requires normalizing scores across BM25 and vector search to the same scale. BM25 scores can be 0–20, cosine scores are 0–1 — directly averaging them without careful normalization gives wrong results. RRF uses only rank positions, not raw scores, so no normalization needed.

### When hybrid search is mandatory

- Any corpus with product codes, IDs, serial numbers, names
- Legal/medical documents with specific terminology
- Code search (function names, variable names, error codes)
- Multi-language corpora where vocabulary overlap is low
- Any production RAG system — hybrid almost always outperforms pure semantic

## Strategy 3: Metadata filtering

### The problem it solves

In a multi-tenant system (multiple users, multiple document sources), semantic search alone retrieves the most similar chunks from the entire corpus. User A could get chunks from User B's private documents. Or a query about 2024 policies returns 2019 policy chunks that are semantically similar but outdated.

Metadata filtering applies hard constraints **before or during** vector search.

### Pre-filtering vs post-filtering

**Pre-filtering (filtered search):** Apply metadata filter first, then run vector search only on the matching subset.

```python
from qdrant_client.models import Filter, FieldCondition, MatchValue

results = qdrant.search(
    collection_name="docs",
    query_vector=query_vector,
    query_filter=Filter(
        must=[
            FieldCondition(key="user_id", match=MatchValue(value="user_123")),
            FieldCondition(key="doc_type", match=MatchValue(value="policy")),
        ]
    ),
    limit=5
)
```

Qdrant runs HNSW only on vectors whose payload matches the filter. Fast, precise, but if the filtered subset is very small, HNSW accuracy degrades (fewer nodes to traverse).

**Post-filtering:** Run vector search first, get top-K, then filter. Simpler but you might filter out most results and end up with fewer than K relevant chunks.

**Qdrant's approach:** Uses a hybrid strategy — if the filter is selective (small subset), it switches to brute-force search on the filtered set. If the filter is broad, it uses HNSW with filtering. Automatic, no config needed.

### What to store in metadata (payload)

```python
payload = {
    "user_id": "user_123",           # multi-tenancy
    "org_id": "indigo_lab37",        # org-level isolation
    "doc_source": "policy_manual",   # document origin
    "created_at": "2024-11-01",      # date filtering
    "doc_type": "faq",               # document type
    "language": "en",                # language filtering
    "access_level": "internal",      # permission level
    "chunk_index": 4,                # position in original doc
    "parent_chunk_id": "chunk_001",  # for parent-child retrieval
}
```

### Multi-tenancy pattern

Never mix tenant data in one collection without filtering — the risk of cross-tenant retrieval is too high.

Two approaches:

```python
# Option 1: One collection per tenant (complete isolation)
# Good for large tenants, higher operational overhead
qdrant.create_collection(f"tenant_{tenant_id}_docs", ...)

# Option 2: Shared collection with tenant_id filter (recommended for many small tenants)
# All tenants in one collection, filter by tenant_id on every query
results = qdrant.search(
    collection_name="all_docs",
    query_vector=query_vector,
    query_filter=Filter(must=[FieldCondition(key="tenant_id", match=MatchValue(value=tenant_id))]),
    limit=5
)
```

For Option 2 to be fast at scale, **always create a payload index** on the filter field:

```python
qdrant.create_payload_index(
    collection_name="all_docs",
    field_name="tenant_id",
    field_schema="keyword"  # exact match index
)
```

Without this index, Qdrant scans all payloads linearly — O(n) on every query.

## Strategy 4: Re-ranking with cross-encoder

### The two-stage retrieval pattern

```
Query
  ↓
Stage 1: Bi-encoder + vector search → top-50 candidates  (fast, approximate)
  ↓
Stage 2: Cross-encoder re-ranking   → top-5 final results (slow, precise)
  ↓
LLM
```

Stage 1 is cheap — one forward pass for the query, dot products against stored vectors.
Stage 2 is expensive but only runs on 50 candidates, not millions.

### Why cross-encoder is more accurate

The bi-encoder never lets the query and document "see" each other. Similarity is geometric — two independently computed vectors compared in space. It's approximate.

The cross-encoder concatenates query + document and runs full self-attention across both. Every query token attends to every document token. It can detect:
- Exact phrase matches
- Negations ("no refund" vs "full refund")
- Subtle relevance signals invisible to geometric comparison

### Implementation

```python
from sentence_transformers import SentenceTransformer, CrossEncoder
from qdrant_client import QdrantClient

bi_encoder = SentenceTransformer("BAAI/bge-large-en-v1.5")
cross_encoder = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")
qdrant = QdrantClient("localhost", port=6333)

def retrieve_and_rerank(query: str, top_k: int = 5) -> list:
    # Stage 1: bi-encoder retrieval
    query_vec = bi_encoder.encode(
        "Represent this sentence for searching relevant passages: " + query,
        normalize_embeddings=True
    )
    candidates = qdrant.search(
        collection_name="docs",
        query_vector=query_vec,
        limit=50   # retrieve more than needed
    )

    # Stage 2: cross-encoder re-ranking
    texts = [hit.payload["text"] for hit in candidates]
    pairs = [[query, text] for text in texts]
    scores = cross_encoder.predict(pairs)

    # sort by cross-encoder score
    ranked = sorted(zip(scores, candidates), key=lambda x: x[0], reverse=True)
    return [hit for _, hit in ranked[:top_k]]
```

### Latency trade-off

| | Bi-encoder only | Bi-encoder + Cross-encoder |
|---|---|---|
| Quality | Medium | High |
| Latency | ~20ms | ~200-500ms |
| Compute | Low | Medium |

The latency hit is real. Mitigations:
- Use a smaller cross-encoder (MiniLM-L-6 instead of L-12)
- Run cross-encoder inference on GPU
- Cache re-ranking results for repeated queries
- Only re-rank when query complexity warrants it

### Popular cross-encoder models

| Model | Speed | Quality | Notes |
|---|---|---|---|
| `cross-encoder/ms-marco-MiniLM-L-6-v2` | Fast | Good | Best latency/quality tradeoff |
| `cross-encoder/ms-marco-MiniLM-L-12-v2` | Medium | Better | Slightly slower |
| `cross-encoder/ms-marco-electra-base` | Slow | Best | Highest quality |
| Cohere Rerank API | API call | Excellent | Managed, no GPU needed |

## Strategy 5: Query expansion

### The vocabulary mismatch problem

A user asks: "how do I get my money back for a delayed flight"

Your document says: "compensation eligibility for flight disruption"

Zero word overlap. Semantic search helps here, but sometimes the embedding space still doesn't bridge the gap — especially for domain-specific terminology the base model hasn't seen enough of.

Query expansion generates multiple phrasings before retrieval, increasing the chance of matching.

### Approach 1: LLM-based expansion

```python
from openai import OpenAI
client = OpenAI()

def expand_query(query: str, n: int = 3) -> list[str]:
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{
            "role": "user",
            "content": f"""Generate {n} different phrasings of this search query.
            Return only the queries, one per line, no numbering.
            
            Original query: {query}"""
        }]
    )
    expansions = response.choices[0].message.content.strip().split("\n")
    return [query] + expansions  # always include original

# Usage
queries = expand_query("money back delayed flight")
# Returns:
# ["money back delayed flight",
#  "refund eligibility for flight delay compensation",
#  "how to claim compensation for delayed flight",
#  "flight disruption reimbursement policy"]

# Retrieve for each, merge results, deduplicate
all_results = []
seen_ids = set()
for q in queries:
    vec = embed(q)
    results = qdrant.search(collection_name="docs", query_vector=vec, limit=10)
    for r in results:
        if r.id not in seen_ids:
            all_results.append(r)
            seen_ids.add(r.id)
```

### Approach 2: Synonym/ontology-based expansion

For controlled domains, maintain a synonym map:

```python
synonyms = {
    "refund": ["money back", "reimbursement", "compensation", "credit"],
    "cancel": ["cancellation", "void", "revoke", "annul"],
    "PNR": ["booking reference", "reservation code", "ticket number"]
}

def expand_with_synonyms(query: str) -> list[str]:
    expansions = [query]
    for term, syns in synonyms.items():
        if term.lower() in query.lower():
            for syn in syns:
                expansions.append(query.lower().replace(term.lower(), syn))
    return expansions
```

### Trade-offs

| | LLM expansion | Synonym expansion |
|---|---|---|
| Coverage | High — creative | Limited to synonym map |
| Latency | +100-300ms (LLM call) | Near zero |
| Cost | LLM API cost per query | None |
| Control | Low | High |
| Maintenance | None | Manual synonym updates |

### When query expansion helps most

- Short, ambiguous queries (1–3 words)
- Domain with rich synonym vocabulary
- Users who don't know exact terminology (support chatbots)
- When recall matters more than precision

### When it hurts

- Query is already specific and long — expansion adds noise
- Latency is critical — extra LLM call adds 100–300ms
- Corpus is small — expansion retrieves duplicates

## Strategy 6: HyDE (Hypothetical Document Embeddings)

### The core insight

Queries and documents live in different regions of vector space — queries are short questions, documents are long answers. Even with asymmetric training, there's a distribution gap.

HyDE bridges this gap by asking: instead of embedding the query, what if we embed a **hypothetical document that would answer the query**? That hypothetical document is stylistically similar to real documents, so it lands much closer to relevant documents in vector space.

```
Normal flow:
Query vector ─────────────────────────────────────────► [vector space]
                                                              ↑ gap
Document vectors ────────────────────────────────────────────┘

HyDE flow:
Query → LLM → Hypothetical answer → embed ──────────────► [vector space]
                                                              ↑ much closer
Document vectors ────────────────────────────────────────────┘
```

### Step by step

```python
from openai import OpenAI
client = OpenAI()

def hyde_retrieve(query: str, qdrant, embedding_model, top_k: int = 5):
    # Step 1: generate hypothetical document
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{
            "role": "user",
            "content": f"""Write a short paragraph that would be the answer to this question.
            Write it as if it's from an official document or knowledge base.
            Do not say 'I' or 'the answer is'. Just write the content directly.
            
            Question: {query}"""
        }]
    )
    hypothetical_doc = response.choices[0].message.content

    # Step 2: embed the hypothetical document, not the query
    hypo_vector = embedding_model.encode(hypothetical_doc, normalize_embeddings=True)

    # Step 3: search using hypothetical document vector
    results = qdrant.search(
        collection_name="docs",
        query_vector=hypo_vector,
        limit=top_k
    )
    return results

# Example
query = "what happens to my booking if indigo cancels the flight"
# Hypothetical doc generated:
# "In the event of a flight cancellation by the airline, passengers are 
#  entitled to a full refund of the ticket fare to the original payment method
#  within 7 business days. Alternatively, passengers may opt for a free 
#  rebooking on the next available flight to the same destination..."
# 
# This reads like a policy document → lands near actual policy chunks in vector space
```

### Why it works

The hypothetical document:
- Uses the same vocabulary as real documents (formal, declarative prose)
- Has similar length and structure to real document chunks
- Doesn't need to be factually correct — it just needs to be stylistically similar to retrieve the right neighborhood in vector space

The factual correctness comes from the retrieved real documents, not the hypothetical.

### When HyDE helps most

- Complex, multi-part questions where the query is far from document style
- Technical domains where users ask casually but documents are formal
- When the query is very short and ambiguous
- Abstract or conceptual questions ("what is the philosophy behind...")

### When HyDE hurts

- Simple factual lookups — the extra LLM call just adds latency
- When the LLM generates a hallucinated hypothetical in the wrong domain
- Low-latency requirements — HyDE adds one full LLM call before retrieval
- Short, specific queries where the query vector already works well

### HyDE vs query expansion — key difference

| | Query expansion | HyDE |
|---|---|---|
| What gets embedded | Multiple query phrasings | One hypothetical answer document |
| Style of embedding input | Query-like (short) | Document-like (long, formal) |
| Number of retrievals | Multiple (one per expansion) | One |
| Best for | Vocabulary mismatch | Query-document style gap |

## Combining strategies — the production retrieval stack

These strategies aren't mutually exclusive. A real production system layers them:

```
User query
    ↓
[1] Query expansion (LLM) → 3 query variants
    ↓
[2] Metadata filtering → restrict to user's tenant + recent docs
    ↓
[3] Hybrid search (BM25 + vector + RRF) → top-50 per variant → merge → top-50 unique
    ↓
[4] Cross-encoder re-ranking → top-5
    ↓
LLM generation
```

HyDE would replace or augment step 1 — instead of or in addition to query expansion.

Not every system needs all layers. Add complexity only when retrieval quality justifies the latency cost:

| System type | Recommended stack |
|---|---|
| Simple internal KB | Semantic search + metadata filter |
| Customer support bot | Hybrid search + metadata filter + re-ranking |
| Enterprise search | All strategies + fine-tuned embedding |
| Real-time chat | Semantic search only (latency priority) |

## Failure modes and fixes

| Symptom | Likely cause | Fix |
|---|---|---|
| Exact product codes never retrieved | Pure semantic search | Add BM25 hybrid search |
| User A seeing User B's results | No metadata filtering | Add tenant_id filter on every query |
| Top-5 irrelevant despite good top-50 | No re-ranking | Add cross-encoder re-ranking |
| Query too vague, nothing relevant returned | Vocabulary mismatch | Add query expansion or HyDE |
| Re-ranking makes things worse | Wrong cross-encoder model | Switch to ms-marco trained model |
| Metadata filter returns 0 results | Filter too restrictive | Debug payload values, check index |
| Hybrid search slower than expected | No sparse vector index | Ensure sparse vectors properly indexed |

## Key interview questions + strong answers

**Q: "When would you use hybrid search over pure semantic search?"**
> When the corpus contains exact identifiers — product codes, IDs, names, error codes — that semantic search can't reliably match. Also whenever users might search with exact terminology that doesn't overlap with document vocabulary. In practice, hybrid search almost always outperforms pure semantic on real-world corpora — the cost is minimal and the coverage improvement is significant. I'd default to hybrid for any production system.

**Q: "Explain RRF and why it's better than weighted average for fusion."**
> RRF scores documents by their rank position across result lists — `1/(k + rank)` — and sums across all lists. Because it uses rank rather than raw score, it requires no normalization between BM25 scores (0–20 range) and cosine scores (0–1 range). Weighted average fusion requires careful score normalization across different scales, which is brittle. RRF is scale-invariant and robust to outlier scores in either list.

**Q: "What's the trade-off with re-ranking?"**
> Re-ranking with a cross-encoder significantly improves precision — the cross-encoder sees full attention between query and document — but adds 200–500ms latency because it can't pre-compute anything. The mitigation is to only re-rank a small shortlist (top-50) and use a fast cross-encoder like MiniLM-L-6. For latency-critical systems, I'd skip re-ranking or run it async and cache results for repeated queries.

**Q: "How does HyDE work and when would you use it?"**
> HyDE generates a hypothetical answer to the query using an LLM, then embeds that hypothetical document instead of the query. The hypothetical document is stylistically similar to real corpus documents — same formal prose, same vocabulary — so it lands closer to relevant documents in vector space than a short query would. I'd use it for complex, multi-part questions or when users ask casually but documents are formal. For simple lookups it's overkill — the extra LLM call adds latency with no benefit.

**Q: "How would you implement metadata filtering in a multi-tenant RAG system?"**
> Every document chunk gets a tenant_id in its payload at index time. Every query includes a mandatory metadata filter on tenant_id — no query runs without it. I'd use a shared collection with payload indexes on tenant_id for efficient filtering, unless tenants are very large (millions of chunks each), in which case separate collections give better isolation. In Qdrant I'd create a keyword payload index on tenant_id to ensure filter lookups are O(log n) not O(n).

## Key terms for interviews

- **BM25** — keyword-based ranking using term frequency and inverse document frequency
- **RRF (Reciprocal Rank Fusion)** — rank-based fusion algorithm; scale-invariant, no normalization needed
- **Hybrid search** — combining BM25 + vector search; covers both exact match and semantic match
- **Metadata filtering** — hard constraints on payload fields before/during vector search
- **Pre-filtering** — filter first, then search on subset; fast but degrades with tiny subsets
- **Post-filtering** — search first, filter results; simple but may return fewer than K results
- **Re-ranking** — cross-encoder scoring on top-K bi-encoder candidates; high precision, high latency
- **Query expansion** — generating multiple phrasings of a query to cover vocabulary mismatch
- **HyDE** — embedding a hypothetical answer document instead of the query to bridge query-document style gap
- **Two-stage retrieval** — bi-encoder for recall, cross-encoder for precision
- **Payload index** — index on a metadata field in Qdrant; required for efficient filtering at scale
- **Sparse vector** — BM25-style high-dimensional sparse representation; used alongside dense vectors for hybrid search
